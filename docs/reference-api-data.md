# API and data reference

The Edge API is the only browser-facing backend. Its committed OpenAPI document is [`contracts/openapi/edge-api.openapi.json`](../contracts/openapi/edge-api.openapi.json), currently containing 62 paths and 83 operations. Service OpenAPI documents describe the private `/internal/*` interfaces.

This page provides the human-readable route, authentication, error, composition, ownership, and migration reference. Use OpenAPI for exact request and response schemas.

## Public route namespaces

| Namespace | Purpose | Default access |
|---|---|---|
| `/health/*`, `/version` | Edge liveness, readiness, and version | Anonymous |
| `/api/auth/*` | Registration, human login/session, API-key exchange | Anonymous endpoint with endpoint-specific credentials |
| `/api/app/*` | Authenticated application API | Human account by default; selected routes permit scoped agents |
| `/api/content/*` | Published educational content | Anonymous read |
| `/api/admin/*` | Content and operations administration | `admin` role |

## Authentication

### Human session

1. `POST /api/auth/login` accepts email/password credentials.
2. Edge stores the returned refresh token in the `td_refresh` HttpOnly cookie.
3. Edge returns only `{ accessToken, expiresAt }` to JavaScript.
4. Authenticated requests send `Authorization: Bearer <access-token>`.
5. `POST /api/auth/refresh` rotates the refresh token and returns a new access token.
6. `POST /api/auth/logout` revokes the refresh family when possible and always clears the cookie.

Registration uses `POST /api/auth/register`. Public registration is disabled by default (`Auth__AllowPublicRegistration=false`). When enabled, the endpoint creates a human `user` account. When public registration is disabled, Identity requires the deployment registration header `X-Registration-Key`; missing or wrong keys return `404` to avoid disclosing the registration surface. Edge rate-limits anonymous auth routes by trusted client IP and returns `429 rate_limited` with `Retry-After` when exceeded.

### Agent access

Agents exchange an API key through `POST /api/auth/api-key/token`. Edge applies route policies:

| Policy | Agent behavior |
|---|---|
| Default application policy | Agents denied; authenticated humans allowed |
| `diaryAccess` | `diary:read` for GET; `diary:write` for mutations |
| `researchAccess` | GET only with `research:read` |
| `admin` | Requires the `admin` role |

API keys are returned once and stored as hashes by Identity. They exchange for short-lived RSA access tokens.

## Headers and transport

| Header | Behavior |
|---|---|
| `Authorization` | Forwarded from Edge to authenticated downstream services |
| `X-Correlation-ID` | Accepted or generated by Edge, forwarded downstream, and returned on the response |
| `Idempotency-Key` | Forwarded for supported Journal create operations; maximum 200 characters |
| `X-Registration-Key` | Opt-in Edge forward only on `POST /api/auth/register` â†’ Identity registration; never on login, refresh, journal, or other proxy routes; never exposed by the browser signup UI |
| `Location` | Downstream location metadata is propagated where applicable |

Edge applies an explicit downstream timeout. The default is 8 seconds and configuration is clamped to 1 through 30 seconds. Browser request cancellation propagates to downstream calls.

### Anonymous authentication rate limits

Partition key is `HttpContext.Connection.RemoteIpAddress` after `UseForwardedHeaders`. Spoofed `X-Forwarded-For` is ignored unless the direct peer is in the configured trusted proxy/network list.

| Route | Policy | Limit |
|---|---|---|
| `POST /api/auth/register` | `auth-register` | 5 requests / 15 minutes / IP |
| `POST /api/auth/login` | `auth-login` | 10 requests / minute / IP |
| `POST /api/auth/api-key/token` | `auth-login` | 10 requests / minute / IP |
| `POST /api/auth/refresh` | `auth-refresh` | 60 requests / minute / IP |

Registration and login request bodies are also capped at 16 KiB on Edge.

## ProblemDetails

Edge errors use `application/problem+json` with this shape:

```json
{
  "code": "validation_failed",
  "title": "Validation failed",
  "status": 422,
  "detail": "The request failed validation.",
  "correlationId": "4a1cd501-7d92-4e3a-97f4-c87fe04746bf"
}
```

| HTTP | Stable code | Meaning |
|---:|---|---|
| 400 | `invalid_request` | Syntax, query, route, or request-body input is invalid |
| 401 | `authentication_required` | Authentication is missing or the session cannot be restored |
| 403 | `access_denied` | The authenticated principal lacks the required role or scope |
| 404 | `resource_not_found` | The resource is missing; cross-owner resources intentionally use the same result |
| 409 | `conflict` | The request conflicts with current state or idempotency history |
| 422 | `validation_failed` | Domain validation failed |
| 429 | `rate_limited` | Anonymous authentication rate limit exceeded; response includes `Retry-After` |
| 502 | `downstream_invalid_response` | A required service returned an invalid typed payload |
| 503 | `downstream_unavailable` | A required service cannot be reached |
| 504 | `downstream_timeout` | A required service exceeded the Edge timeout |

Responses never include stack traces, SQL, internal URLs, credentials, or downstream raw error bodies.

### Market data and price alerts

| Methods | Path |
|---|---|
| GET | `/api/app/market/symbols` |
| GET | `/api/app/market/bars/{symbol}` |
| GET | `/api/app/market/providers/health` |
| GET, POST | `/api/app/price-alerts` |
| PUT, DELETE | `/api/app/price-alerts/{id}` |
| POST | `/api/app/price-alerts/{id}/dismiss` |
| POST | `/api/app/price-alerts/{id}/reactivate` |
| GET | `/api/app/price-alerts/{id}/triggers` |

Price alerts are evaluated from the read-only `market_data_public.daily_bar_prices_v1` contract only after a daily bar is published. `evaluationPrice` and trigger `priceType` are lowercase `open | close` enums and default to `close`; alert `status` is `active | triggered | dismissed`. Open means the opening value inside the completed daily bar, not a real-time market-open quote. Trigger responses retain `observedClose` for compatibility and also identify the actual `observedPrice` and `priceType`.

Only active alerts whose latest published trade date is newer than `lastEvaluatedDate` consume a worker batch slot. A trade date is evaluated at most once per alert, and concurrent workers lock only claimed alert rows. Dismissing a triggered alert timestamps its latest active trigger; dismissing an active alert only pauses evaluation. Reactivation preserves `lastEvaluatedDate` and waits for a newer bar.

Market ingestion is not browser-facing. An external authenticated loader uses the Market Data service-key admin flow documented in [`services/market-data-service/SERVICE.md`](../services/market-data-service/SERVICE.md).

### Market rotation

| Methods | Path |
|---|---|
| GET, POST | `/api/app/rotation/universes` |
| PUT, DELETE | `/api/app/rotation/universes/{id}` |
| PUT | `/api/app/rotation/universes/{id}/symbols` |
| POST | `/api/app/rotation/universes/{id}/calculate` |
| GET | `/api/app/rotation/monitor` |

The monitor returns stored backend calculations. `universe.rankScope` identifies whether `rank2w` is global to the universe (`universe`) or local to each sector (`sector`). Each row's `rankGroup` identifies that rank partition: the universe code for universe scope, or the sector name/`Unclassified` for sector scope. Sector-local ranks must not be compared across rank groups.

The raw `percentile2w` contract range is `0` through `1`; clients may format it as `0%` through `100%` without recalculating it. Missing lookback data remains `null` with an appropriate status. The frontend does not calculate returns, moving averages, rank, percentile, breadth, or market state.

### Partners, content, tools, and operations

## BFF composition semantics

### Capability status

Optional capabilities use exactly:

- `available`: the optional service returned usable data.
- `empty`: the service succeeded but the domain value is absent where absence is valid.
- `unavailable`: the optional service failed without an authorization failure.

### Dashboard

Required inputs are Journal diary-day facts, the diary collection, and Performance. Missing daily Performance is a valid `null`. Discipline and Diary Alerts are optional, but their `401` or `403` still fails the complete request.

### Calendar

Journal facts, daily Performance records, and the monthly Performance summary are required. Diary Alerts are optional. Edge creates one item for every calendar date and preserves missing P/L as `null`.

### Stock page

Stock Research is required. Market Data bars are optional, so a valid stock page can return with capability `unavailable` and `bars: null`.

## Events

The current domain event is `DiaryDeleted.v1`. Journal writes it through a transactional outbox; Reminder consumes it through an inbox. The envelope schema is [`contracts/events/event-envelope.schema.json`](../contracts/events/event-envelope.schema.json) and the payload schema is [`diary-deleted.v1.schema.json`](../contracts/events/diary-deleted.v1.schema.json).

## Migration contract

`platform/postgres/migrations/manifest.json` currently contains migrations `0001` through `0017`. Migration files are append-only and checksummed. The legacy baseline is fixed through `0013`; `0014` and later are normal pending migrations after baseline adoption.

Use the commands and production adoption sequence in [Database migrations](database-migrations.md). Never edit an applied SQL file or a history row.

## Null and absence rules

- Missing daily P/L is `null`, never zero.
- Missing score averages are `null`; absent scores do not enter the average denominator.
- Missing market lookback values remain `null` and normally carry `insufficient_data`.
- Missing and cross-owner resources both return `404`.
- A timeout, unavailable dependency, or invalid payload is not a valid empty response.

## Contract maintenance

Generate and verify contracts with:

```sh
npm --prefix frontend run api:generate
npm --prefix frontend run api:verify
python scripts/validate-openapi.py
```

`api:generate` updates service OpenAPI, composed Edge OpenAPI, and the generated frontend client. Review source DTO and endpoint changes together with generated diffs. Never edit generated client output manually.
